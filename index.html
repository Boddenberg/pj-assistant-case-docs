🔀 Fluxo de uma Request

  1. Cliente faz POST /v1/chat/{customerId}
  2. BFA (Go) recebe a request e valida autenticação JWT
  3. Concorrentemente (errgroup): busca Profile + Transactions
  4. Monta payload e chama POST /v1/chat no Agente
  5. Agente valida input (sanitização, injection, PII masking)
  6. LangGraph: Planner → Tools → Synthesizer
  7. Retorna resposta estruturada com métricas de tokens/custo
  8. BFA registra métricas Prometheus e retorna ao cliente

📐 Decisões Arquiteturais

  • BFA em Go: Performance para I/O concorrente, type safety, deploy leve (~8MB binary)
  • Agent em Python: Ecossistema de IA (LangChain, LangGraph, ChromaDB)
  • Arquitetura Hexagonal: Domain puro, ports como interfaces Go, adapters plugáveis
  • Contrato Versionado (v9): BFA e Agent independentes, evoluem separadamente
  • Deploy Railway: CI/CD automático no push (Dockerfile multi-stage)
  • Supabase (PostgREST): Zero ORM, PostgreSQL puro com RLS

📄 Resilience Pattern — Retry + Circuit Breaker + Bulkhead 
// infra/resilience/resilience.go

func RetryWithBackoff(ctx context.Context, cfg Config, fn func() error) error {
    var lastErr error
    for attempt := 0; attempt <= cfg.MaxRetries; attempt++ {
        if err := ctx.Err(); err != nil { return err }
        lastErr = fn()
        if lastErr == nil { return nil }
        backoff := time.Duration(math.Pow(2, float64(attempt))) * cfg.InitialBackoff
        jitter := time.Duration(rand.Int63n(int64(backoff / 2)))
        select {
        case <-ctx.Done(): return ctx.Err()
        case <-time.After(backoff + jitter):
        }
    }
    return lastErr
}

func NewCircuitBreaker(name string) *gobreaker.CircuitBreaker {
    return gobreaker.NewCircuitBreaker(gobreaker.Settings{
        Name:        name,
        MaxRequests: 3,               // half-open: allow 3 probes
        Interval:    30 * time.Second, // reset counters every 30s
        Timeout:     10 * time.Second, // open → half-open after 10s
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
            return counts.Requests >= 5 && failureRatio >= 0.6
        },
    })
}

graph = StateGraph(AgentState)

# ─── Registrar nós
graph.add_node("planner", planner_node)       # Nó 1: Planejar (LLM + bind_tools)
graph.add_node("executor", executor_node)     # Nó 2: Dispatcher

Trade-offs & Visão de Produção

⚖️ Trade-offs Assumidos

  • ChromaDB local vs Pinecone/Weaviate → Zero infra para MVP, rápido para iterar. Em prod: managed vector DB.
  • OpenAI Embeddings vs Sentence Transformers local → Imagem Docker 3x menor (~500MB vs ~1.5GB). Custo desprezível ($0.02/1M tokens).
  • Respostas determinísticas para onboarding vs LLM → Zero hallucination, zero tokens, latência mínima, 100% consistente.
  • Railway vs AWS → Deploy em 1 min, zero config. Prod real: ECS + ALB + RDS.
  • Regex para prompt injection vs ML classifier → Pega ~80% dos ataques. Prod real: NeMo Guardrails ou classifier dedicado.

🚀 O que faria em Produção Real

  • PII Detection robusto: Microsoft Presidio ou AWS Comprehend
  • Guardrails: NeMo Guardrails para controle de escopo e segurança
  • A/B Testing de prompts: LangFuse com variantes e métricas por versão
  • Vector DB managed: Pinecone ou pgvector em RDS
  • LLM-as-judge: Avaliação automatizada de qualidade com GPT-4
  • Event-driven: SQS/SNS para notificações e processamento assíncrono
  • AWS: ECS Fargate + ALB + RDS + ElastiCache + CloudWatch
  • Reranking: Cross-encoder ou Cohere Rerank para melhorar precisão RAG
