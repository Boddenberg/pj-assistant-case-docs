Fluxo de uma Request

  1. Cliente faz POST /v1/chat/{customerId}
  2. BFA valida autenticação JWT
  3. Concorrentemente (errgroup): busca Profile + Transactions
  4. Monta payload e chama POST /v1/chat no Agent
  5. Agent valida input (sanitização, injection, PII masking)
  6. LangGraph: Planner → Tools → Executor → Synthesizer
  7. Retorna resposta com reasoning, tokens e custo estimado
  8. BFA registra métricas Prometheus e retorna ao cliente

Decisões Arquiteturais

  • BFA em Go: Performance para I/O concorrente, type safety, binary ~8MB
  • Agent em Python: Ecossistema de IA — LangChain, LangGraph, ChromaDB
  • Hexagonal: Domain puro, ports como interfaces, adapters plugáveis
  • Contrato v9: BFA e Agent independentes, evoluem separadamente
  • Railway: CI/CD auto-deploy no push (Dockerfile multi-stage)
  • Supabase: Zero ORM, PostgreSQL com RLS, PostgREST

Agente de IA Generativa

O coração da solução. Grafo LangGraph com 4 nós, 3 tools especializadas, RAG pipeline completo e camada de segurança multi-layer. Cada componente foi projetado para ser auditável, testável e production-ready.

LangGraph Workflow — Grafo de Execução

O agente usa StateGraph(AgentState) com 4 nós em um grafo dirigido. O Planner recebe o contexto do cliente (perfil + transações + histórico de conversa) e decide quais tools chamar via bind_tools(). O Executor processa os resultados e pode iterar (loop) se mais tools forem necessárias. O Synthesizer consolida tudo em uma resposta final conversacional, sem formato de relatório. A função should_continue() implementa o roteamento condicional — se a última AIMessage contém tool_calls, vai para Tools; senão, sintetiza.

Planner Node

Planejamento Inteligente

LLM com bind_tools(AGENT_TOOLS) decide autonomamente quais ferramentas invocar. Recebe System Prompt v9 + contexto financeiro do cliente. Contabiliza tokens via usage_metadata.

Tool Calling

3 Tools Especializadas

analyze_transactions — análise financeira. search_knowledge_base — RAG semântico. assess_credit_profile — avaliação de crédito. Tools nunca lançam exceção (retornam erro como string).

RAG Pipeline

Retrieval-Augmented Generation

ChromaDB + text-embedding-3-small (OpenAI, 1536 dims). Chunking 1024 chars com overlap 128. Threshold de similaridade para filtrar ruído. 10+ documentos de knowledge base.

Synthesizer

Resposta Contextualizada

Consolida análises em resposta conversacional. Suporta instrução de onboarding dinâmica. Retorna metadata estruturada: reasoning steps, fontes RAG, tokens, custo estimado.

Security

4 Camadas de Proteção

Validação de input (vazio, tamanho). Detecção de prompt injection (regex PT/EN). PII masking (CPF, CNPJ, cartão, email) antes do LLM. Circuit breaker de custo ($0.10/request).

Onboarding

State Machine Determinística

10 campos campo-a-campo, sem LLM (zero hallucination, zero tokens). Validação inline de formato. Retomada de sessão via history. Contrato v9 com step/field_value/next_step.

Observability

Rastreabilidade Completa

structlog com JSON + Axiom. Prometheus (latência, tokens, custo, fallback). OpenTelemetry tracing. LangFuse para traces de LLM. Cada step do AgentState registrado.

Prompt Engineering

System Prompt v9

Versionamento semântico em prompts.py. GPT-4o-mini com temperature 0.1. Instruções anti-hallucination: "NUNCA invente dados". Guardrail: apenas assuntos bancários PJ.

Evaluation

LLM-as-Judge

GPT-4 avalia conversas em 9 critérios ponderados (correctness 20%, faithfulness 20%, safety 15%). Endpoint POST /v1/evaluate. Threshold 7.0 para PASS. Retorna JSON com scores, reasoning e improvements acionáveis.

# agent/graph.py — 411 linhas, o "cérebro" do agente

graph = StateGraph(AgentState)

# Registrar nós
graph.add_node("planner", planner_node)       # LLM + bind_tools → decide tools
graph.add_node("executor", executor_node)     # LLM pós-tool → pode iterar
graph.add_node("tools", tool_node)             # ToolNode auto-executa tools
graph.add_node("synthesizer", synthesizer_node) # LLM → resposta final

# Arestas
graph.set_entry_point("planner")
graph.add_conditional_edges("planner", should_continue,
    {"tools": "tools", "synthesize": "synthesizer"})
graph.add_edge("tools", "executor")
graph.add_conditional_edges("executor", should_continue,
    {"tools": "tools", "synthesize": "synthesizer"})  # loop possível
graph.add_edge("synthesizer", END)

# Roteador condicional
def should_continue(state: AgentState) -> str:
    last = state["messages"][-1]
    if hasattr(last, "tool_calls") and last.tool_calls:
        return "tools"
    return "synthesize"

// infra/resilience/resilience.go

func RetryWithBackoff(ctx context.Context, cfg Config, fn func() error) error {
    var lastErr error
    for attempt := 0; attempt <= cfg.MaxRetries; attempt++ {
        if err := ctx.Err(); err != nil { return err }
        lastErr = fn()
        if lastErr == nil { return nil }
        backoff := time.Duration(math.Pow(2, float64(attempt))) * cfg.InitialBackoff
        jitter := time.Duration(rand.Int63n(int64(backoff / 2)))
        select {
        case <-ctx.Done(): return ctx.Err()
        case <-time.After(backoff + jitter):
        }
    }
    return lastErr
}

func NewCircuitBreaker(name string) *gobreaker.CircuitBreaker {
    return gobreaker.NewCircuitBreaker(gobreaker.Settings{
        Name:        name,
        MaxRequests: 3,               // half-open: allow 3 probes
        Interval:    30 * time.Second, // reset counters every 30s
        Timeout:     10 * time.Second, // open → half-open after 10s
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
            return counts.Requests >= 5 && failureRatio >= 0.6
        },
    })
}

Trade-offs & Visão de Produção

Trade-offs Assumidos

  • ChromaDB local vs Pinecone/Weaviate→ Zero infra para MVP. Em prod: managed vector DB.
  • OpenAI Embeddings vs Sentence Transformers→ Imagem Docker 3x menor (~500MB vs ~1.5GB). Custo: $0.02/1M tokens.
  • Onboarding determinístico vs LLM→ Zero hallucination, zero tokens, latência mínima.
  • Railway vs AWS→ Deploy em 1 min, zero config. Prod real: ECS + ALB + RDS.
  • Regex p/ prompt injection vs ML classifier→ Pega ~80% dos ataques. Prod: NeMo Guardrails ou classifier.

Em Produção Real

  • PII Detection: Microsoft Presidio ou AWS Comprehend
  • Guardrails: NeMo Guardrails para controle de escopo
  • A/B Testing: LangFuse com variantes e métricas por versão
  • Vector DB: Pinecone ou pgvector em RDS
  • LLM-as-judge: Avaliação automática com GPT-4
  • Event-driven: SQS/SNS para processamento assíncrono
  • AWS: ECS Fargate + ALB + RDS + CloudWatch
  • Reranking: Cross-encoder ou Cohere Rerank

Experiência para Avaliação

O case foi construído como um aplicativo mobile funcional — uma experiência bancária completa que permite explorar, testar e avaliar a solução de ponta a ponta.

Por que um app funcional?

Em vez de entregar apenas código e documentação, o case entrega uma experiência interativa. O avaliador pode baixar o app, navegar como um cliente PJ e testar o chat conversacional com dados reais — tudo alimentado pelo backend Go e pelo agente de IA.

A proposta é tornar a avaliação tangível: cada botão do app dispara um endpoint real do BFA, cada pergunta no chat passa pelo pipeline completo do agente LangGraph.

App Bancário

Interface mobile com navegação real: PIX, contratação de cartão de crédito, extrato, fatura, análise financeira e PIX via cartão de crédito.

DevTools Embutidas

Ferramentas de simulação dentro do app: adicionar saldo, ajustar limite de crédito, gerar transações e compras no cartão — monte cenários em segundos.

Chat Conversacional

Pergunte sobre transações, gastos e comportamento financeiro. O agente de IA analisa dados reais e responde com contexto. Inclui jornada de abertura de conta pelo chat.
